# 极简前端重规划（对齐赛题与评审）

## 1. 规划基线（来自题目要求）

根据 `docs-md/03-需求/题目.md` 的要求，前端必须服务于 4 个评分维度：

- **作品完整性（40）**：必须体现接处警、目标布控、预警复核的业务闭环。
- **应用创新性（25）**：要展示“实战效率提升”，而不是只做静态展示页。
- **技术创新性（20）**：前端要可解释地呈现 RAG、多源信息、推理过程关键节点。
- **系统性能（15）**：要可视化响应时延、处理吞吐与关键操作耗时。

结论：前端不再只是“简洁好看”，而是“**极简 + 闭环可证 + 指标可评**”。

---

## 2. 新前端目标（比赛导向）

### 2.1 一句话目标

做一个没有装饰噪音、可以 3 分钟完整演示“输入警情 -> 风险研判 -> 布控建议 -> 人工复核 -> 导出留痕”的实战工作台。

### 2.2 视觉约束（强制）

- 无 emoji
- 无渐变
- 无花哨动效
- 颜色只表达状态，不做装饰
- 版面按“重点优先”排序，不按“功能多少”堆叠

---

## 3. 页面重构（由 2 页升级为 3 区）

## Page A：实战工作台（主流程页，必须首屏可闭环）

> 对应评分：完整性、应用创新性、系统性能

模块顺序固定：

1. **警情输入区**
   - 报警文本输入
   - 示例一键填充
   - 开始研判按钮

2. **研判结果区（P0）**
   - 警情类型
   - 风险等级
   - 处置建议
   - 核心要素（地点/人数/武器/伤情/紧迫性）

3. **目标布控区（P0）**
   - 嫌疑人特征摘要
   - 布控点位
   - 盘查要点
   - 缺失信息提示

4. **预警复核区（P0）**
   - 复核结论（同意/驳回/待补充）
   - 复核备注
   - 复核记录导出

5. **操作留痕区（P1）**
   - 研判结果导出（JSON/Markdown）
   - 布控方案导出（Markdown）

---

## Page B：证据与可解释页（评审展示页）

> 对应评分：技术创新性、完整性

仅承载“评委要看但一线首屏不需要”的内容：

- RAG 检索片段（折叠）
- 推理中间关键信息（折叠）
- 原始结构化输出（折叠）
- 异常与兜底说明（折叠）

要求：每个技术块都附 1 句“业务意义”，避免纯技术堆栈展示。

---

## Page C：性能与成效页（答辩支撑页）

> 对应评分：系统性能、应用创新性

固定看板：

- 核心吞吐：总处理量、近 10 次平均耗时
- 风险分布：紧急/高/中/低
- 闭环质量：复核通过率、需补充信息率
- 业务价值：平均处置路径步数、回填重跑成功率

要求：所有指标可回溯，不展示“无法解释来源”的数字。

---

## 4. 信息分层规范（严格执行）

- **P0 首屏必显**：类型、风险、建议、布控、复核动作
- **P1 次级可见**：关键字段详情、导出
- **P2 折叠查看**：RAG 片段、原始 JSON、异常文本
- **P3 独立页面**：统计与性能看板、历史趋势

---

## 5. 演示链路（3 分钟模板）

1. 输入口语化警情，点击研判（展示语义理解能力）  
2. 展示风险级别与处置建议（展示决策支持）  
3. 切到布控区生成方案（展示目标溯源能力）  
4. 完成人工复核并导出（展示业务闭环）  
5. 切到性能页展示时延与准确相关指标（展示工程可用性）

---

## 6. 前端实现约束（与当前代码兼容）

- `web/index.html` / `web/app.js` 仅做页面编排与交互
- `web/styles.css` 仅做布局与样式
- `src/services/*` 仅做业务处理与数据组装
- 所有“文案/导出内容/规则判断”放服务层，不放 UI 组件

---

## 7. 两周落地排期（建议）

### 第 1 周：结构闭环

- D1-D2：把主流程页补齐“布控 + 复核 + 导出”闭环
- D3-D4：新增证据与可解释页
- D5：联调并固定演示链路

### 第 2 周：评分强化

- D6-D7：新增性能与成效页指标
- D8-D9：统一视觉 token（字体、间距、边框、状态色）
- D10：全流程压测与答辩彩排

---

## 8. 验收标准（按评分映射）

- **完整性**：单页可跑通“研判-布控-复核-导出”闭环
- **应用创新性**：能量化展示效率提升（时长、步数、干扰率）
- **技术创新性**：能解释 RAG/推理链路与业务价值对应关系
- **系统性能**：可稳定展示响应时延与风险统计，且数据来源清晰
