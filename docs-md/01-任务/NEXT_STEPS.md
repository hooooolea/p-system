# 警擎项目 · 完善路线图（下一步做什么）

> 依据：`question.md`（榜题 07 命题与评审标准）、`README.md`（当前架构与依赖）  
> 当前状态：已有可运行工作台（HTML + Flask + LLM + RAG），需向「完整作品」演进。

---

## 一、技术选型结论（不用再从零选框架）

| 层级 | 建议 | 说明 |
|------|------|------|
| 编排与 RAG | **继续用 LangChain（langchain + langchain-community + langchain-core）** | 项目已用 `ChatZhipuAI`、向量库 `Chroma`、嵌入 `ZhipuAIEmbeddings`；换框架成本高，除非有强理由。 |
| 向量库 | **Chroma**（可逐步迁移到 `langchain-chroma` 官方包，消除弃用告警） | 满足 demo；数据量大再评估 Milvus / PGVector。 |
| 前端演示 | **HTML + `api_server.py`** | 主演示入口；可选 `app/` 子工程另建生产级 SPA。 |
| 大模型 | **智谱 GLM（当前 `glm-4-flash`）** 或 **通义（`LLM_PROVIDER=dashscope`）** | 以 `.env` 为准；评测时固定一个 provider，避免指标不可比。 |

结论：**LangChain + Chroma + HTML 工作台 + 一家 LLM API** 作为主线即可；完善阶段重点是「业务闭环 + 数据与评测 + 文档」，而不是换栈。

## 三、分阶段任务清单（建议按顺序推进）

### 阶段 A：工程与可复现（1～2 天）

- [ ] **依赖锁版本**：`requirements.txt` 里对关键包补上限或 `pip freeze` 出一份 `requirements.lock`（答辩环境可复现）。
- [ ] **环境与密钥**：只在本机保留 `.env`，**勿把真实 Key 写入仓库**；`env.example` 仅保留占位符。
- [ ] **README 与真实目录一致**：核对 `README.md` 中 `cd …`、`env.example` 等路径与仓库实际一致（避免评委照抄跑不通）。
- [ ] **一键自检**：保留并完善 `python test_quick.py`；可选增加 `pytest` 对解析/兜底逻辑的单元测试（不依赖外网时可 mock）。

### 阶段 B：核心业务闭环（对齐 40 分「完整性」）

- [ ] **接处警研判**：结构化输出稳定（类型、风险、摘要、建议、法条）；风险规则与 Prompt 可维护（配置或单独模块）。
- [ ] **目标布控（Tab2）**：从「一段生成文字」升级为**可结构化输出**（特征摘要、卡点、通报模板），并保存为可导出文本/Markdown。
- [ ] **溯源/预警复核（至少一种可演示）**：例如「低置信度警情 → 提示人工复核要点」或「布控要素缺失清单」，体现“预警复核”能力。
- [ ] **RAG 知识库**：`knowledge_base/` 增加真实可引用的规程片段（注意合规与脱敏）；建立**类目标签**便于检索质量评估。

### 阶段 C：数据与评测（对齐 15 分「系统性能」+ 报告数据）

- [ ] **构建测试集**：如 README 所写，收集 **≥50 条** 警情样例（脱敏），标注类型与风险。
- [ ] **跑分脚本**：输出准确率、平均耗时、失败样例列表（写入 `data/`）。
- [ ] **Chroma 弃用告警**：评估迁移到 `langchain-chroma`（非必须，但能减少技术债说明）。

### 阶段 D：创新与扩展（按需加分，控制范围）

- [ ] **多模态/视频方向（命题提到）**：先做「方案 + 小规模 demo」即可——例如：上传图片 → 用多模态 API 生成文字特征 → 再接入布控模块（不必一上来自建 CLIP）。
- [ ] **云边协同**：可作为架构图与文字设计（边缘采集、中心推理），代码可用「异步队列/批处理」示意。

### 阶段 E：提交物（命题要求）

- [ ] **作品设计报告**（问题背景、方案、架构、创新点、局限与展望）。
- [ ] **技术实现说明**（模块说明、依赖、运行步骤、评测方法）。
- [ ] **源代码**：确保可运行；附 `NEXT_STEPS.md` 或 README 指向本路线图，方便评委理解迭代计划。
