# 警擎前端架构说明

本文描述当前与目标前端架构：在**可落地**的前提下，对齐**中国移动政企/门户类**常见视觉与信息层级，并借鉴 [焕白·本草志展示站](https://goodface.pages.dev/) 的**页面骨架与分区叙事**（非化妆品业务与装饰风格）。

---

## 1. 对标关系（写什么、不写什么）

| 维度 | 借鉴 [goodface.pages.dev](https://goodface.pages.dev/) | 中国移动风格（政企/能力门户常见） |
|------|----------------------------------------------------------|-------------------------------------|
| **结构** | 顶栏主导航 → 首屏标题区 → 多段 `section` 纵向展开 → 页脚辅助信息 | 顶栏品牌 + 主导航；内容区卡片化；表格/列表承载规范信息 |
| **导航** | 横向文字导航（首页 / 产品 / …） | 同级不超过 5～7 项；当前页高亮；可选页内锚点 |
| **分区** | 每段有「小标题 + 说明 + 内容块」节奏 | 模块标题左对齐 + 分隔线；次要信息折叠或二级页 |
| **不照搬** | 视频/海报/动效-heavy、营销渐变 | 避免花哨动效与强营销感；**无 emoji、无渐变**（与赛题演示一致） |

## 2. 信息架构（IA）

```
站点根 /
├── /                    工作台（顶栏主导航 + 左侧竖向「站点」条 + 主栏接警/研判 + 右侧本页目录）
├── /judgments           研判记录总览（左列表 + 右专页同款 presentation）
├── /performance         性能统计（独立页）
├── /analysis-result     研判结果专页（可选）
└── /tech-route          技术路线（静态说明页）
```

**布局参考（与早期工作台一致）**：`doc-layout` 为 **三栏**——左 `doc-sidebar`（竖向「站点」：工作台 / 研判记录 / 性能统计 / 技术路线）、中 `doc-main`、右 `doc-aside`（本页锚点）。语音与文字接警之间用 **`shell-divider`** 分隔，不用整块外包一层。

**原则**

- **工作台主闭环**在 `/`；性能统计在 `/performance`；研判记录总览在 `/judgments`。
- **状态可见**：工具带保留「检索增强」与轻量会话指标。

## 3. 页面布局范式（模仿 goodface 的「骨架」）

以下与 [goodface.pages.dev](https://goodface.pages.dev/) 同构的是**信息顺序**，不是皮肤。

### 3.1 单页通用骨架

```
┌─────────────────────────────────────────────────────────┐
│ site-header（吸顶）                                       │
│  左：产品/项目名 + 一行定位语                               │
│  右：主导航链接（当前项 active）                            │
├─────────────────────────────────────────────────────────┤
│ site-toolbar（可选，一行）                                │
│  状态 chip、环境开关、轻量提示                             │
├─────────────────────────────────────────────────────────┤
│ site-main（垂直 stack，段间距统一）                       │
│  section.panel × N                                        │
│    ├─ panel-head：区块标题 + 右侧次要操作                  │
│    └─ body：表单 / 指标卡 / 表格 / 折叠区                 │
├─────────────────────────────────────────────────────────┤
│ site-footer（可选）                                       │
│  版本、文档链接、赛题编号、免责声明                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 与工作台的对应

| goodface 区块意象 | 警擎工作台落地 |
|-------------------|----------------|
| 首屏标题 + CTA | 顶栏品牌区 +「接警输入」主按钮（研判） |
| 多段「亮点 / 数字」 | 「研判结果」指标卡 + 关键信息栅格 |
| 「定制」多步表单 | 接警文本 → 布控多字段（可后续改为步骤条） |
| 页脚导航 | 文案指向仓库内 Markdown（专网说明等），不依赖已删除的独立静态页 |
| 页内横向索引 + 纵向大段 | 每个 **标签页** 内再设 `inpage-nav` 锚到 `#sec-*`；各段为 `section.panel.section-block`（左色条 + `block-kicker` 小标题） |

### 3.3 长内容

- 各标签页内仍用 **多个 `section.panel` / 锚点** 垂直展开；表格用 `simple-table`、`grid-table` 等现用类名。

## 4. 中国移动风格设计系统（Design Tokens）

政企与移动系门户常见取向：**蓝主色、白底、灰阶辅助、少装饰**。建议在 `styles.css` 的 `:root` 中统一维护（可与当前变量对齐并逐步替换）。

### 4.1 色彩（建议值）

| Token | 用途 | 建议值 |
|-------|------|--------|
| `--cm-blue` | 主色、主按钮、链接高亮 | `#0085D0`（移动蓝参考） |
| `--cm-blue-dark` | hover / 强调 | `#006BA6` |
| `--cm-bg-page` | 页面底 | `#F2F4F7` |
| `--cm-surface` | 卡片背景 | `#FFFFFF` |
| `--cm-border` | 分割线、表框 | `#D9DEE8` |
| `--cm-text` | 主文 | `#1A1A1A` |
| `--cm-text-secondary` | 辅文 | `#5C6B7A` |
| `--cm-danger` | 紧急/错误（克制使用） | `#C62828` |

**红色**：移动品牌红多用于 Logo/强调条，全站大面积红易「政务告警」感过重；建议**仅顶栏底边 2px 或 active 下划线**点缀，而不用渐变底。

### 4.2 字体与排版

- **中文**：`PingFang SC` / `Microsoft YaHei` / `Noto Sans SC`（与现有一致）。
- **字号阶梯**：12 / 13 / 14 / 16 / 18（页面标题），避免超过 5 级。
- **行长**：正文 `max-width: 68ch`～`72ch`（已实现 `content-narrow`），防止换行「锯齿感」。
- **圆角**：卡片 10～12px；按钮与输入框一致，避免混用多种圆角。

### 4.3 组件层级（中国移动常见「整齐感」来源）

1. **顶栏**：白底 + 底边框 + 轻阴影；高度固定，内容垂直居中。
2. **卡片**：白底、细边框、极轻阴影；标题区与内容区 **一条分隔线**。
3. **主按钮**：满宽或固定最小宽度，**单页仅 1 个主色按钮**（当前：研判）。
4. **表格**：表头浅灰底、行 hover 极浅灰，**禁止**花边框。

## 5. 技术架构（当前实现）

```
浏览器
  └── 静态资源（Flask 托管 web/）
        ├── index.html      单页工作台壳 + 语义结构
        ├── styles.css      全局 Design Tokens + 布局 + 组件
        └── app.js          工作台：API 调用、DOM 更新、下载、评审维度表（读 performance.scorecard）

  └── HTTP API（api_server.py）
        ├── GET  /
        └── JSON /api/*（含 performance 内嵌 scorecard）
```

**特点**

- **无构建步骤**：适合赛题演示与快速迭代；后续可迁到 Vite + 组件化而不改接口。
- **前后端分离**：UI 只通过 `fetch('/api/...')` 与后端交互；业务规则在 `src/services/`。

## 6. 推荐目录演进（可选下一阶段）

若项目扩大，建议迁为：

```
frontend/
  public/
  src/
    styles/
      tokens.css          # 移动风变量
      layout.css          # site-header / page-wide / stack
      components/         # card, table, form, nav
    pages/
      workbench.html      或 Vue/React 路由页
    api/
      client.js           统一 fetch + 错误处理
```

当前保持 `web/` 单目录亦可，只要把 **tokens / layout / components** 在逻辑上分层（可先拆为 `tokens.css` + `layout.css` 多文件，由 `index.html` 链入）。

## 7. 与后端的契约（摘要）

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | `/api/analyze` | 接警研判 |
| GET | `/api/history` | 历史列表 |
| POST | `/api/review` | 复核保存 |
| POST | `/api/bukong` | 布控方案 |
| GET | `/api/performance` | 性能统计 + `scorecard`（评审四维映射与布控计数） |

前端 **不拼接业务字符串用于展示关键字段** 时，应对用户/模型返回文本做转义（当前 `app.js` 已 `esc()`）。
